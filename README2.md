# README2 — идеи развития продукта

## Вердикт по проекту — 2026-05-15

### Коротко

Проект сейчас уже не выглядит как сырой генератор задач. Ядро стало сильным: есть 7 языков, теория и практика связаны, task generator покрыт тестами, backend собирается, серверные зависимости после последних правок чистые, появились onboarding, skill map, цели, progress report, AI hints/breakdown и интеграционный тестовый контур.

Главный вывод: **нужно не добавлять еще больше всего, а довести учебный цикл до ощущения личного тренера**. Самая ценная петля продукта:

**диагностика → тренировка дня → ошибка как урок → повторение → карта роста → отчет прогресса.**

Если эта петля будет ощущаться надежной, спокойной и полезной каждый день, продукт будет нужен людям без агрессивной монетизации.

### Что сейчас в хорошем состоянии

- Теория/практика: контракт прошел для 94 тем по 7 языкам.
- Генератор задач: smoke-тест прошел на 200 задачах.
- Контракты задач: 7 файлов прошли проверку.
- Runtime timeout: прошел.
- Backend build: `tsc` прошел.
- Backend audit: `npm audit --omit=dev --audit-level=moderate` показал `0 vulnerabilities`.
- Leaderboard уже имеет server-side Pro enforcement.
- Progress sync уже получил `clientUpdatedAt` и 409 conflict flow.
- Electron security база нормальная: `contextIsolation: true`, `nodeIntegration: false`.

### Что нужно сделать в первую очередь

#### 1. Убрать refresh token из renderer localStorage

Это главный технический риск. Сейчас refresh token лежит в renderer:

- `studingJS/src/renderer/api/auth.mjs`
- `studingJS/src/renderer/api/client.mjs`

Даже при `contextIsolation` лучше хранить refresh/session в main process или secure storage, а renderer должен просить действия через IPC. Это нужно сделать до серьезного публичного релиза.

#### 2. Починить frontend dependency audit

`studingJS` все еще имеет moderate audit issue через:

- `monaco-editor`
- `dompurify`

`npm audit fix --force` откатывает Monaco на breaking-версию, поэтому делать вслепую не надо. Нужен отдельный подход:

- проверить, есть ли безопасная версия Monaco выше текущей;
- если нет, оценить реальный exploit surface в Electron;
- зафиксировать risk note до обновления upstream.

#### 3. Сделать интеграционные тесты удобными и честными

`server/src/tests/integration.test.ts` уже хороший шаг: он покрывает auth, progress, custom tasks, leaderboard plan enforcement. Но текущий `npm run test:integration` падает, если локальный сервер не запущен.

Нужно сделать один из вариантов:

- `test:integration:local` сам поднимает сервер и тестовую БД;
- `test:integration:staging` бьет только в отдельный staging backend;
- prod-тесты не создают мусорных пользователей или умеют чистить за собой.

Иначе тест есть, но его легко неправильно прогнать и получить шум.

#### 4. Добавить privacy/consent для AI hints и breakdown

AI routes отправляют условие задачи и код пользователя во внешний AI provider. Это полезная фича, но пользователь должен понимать:

- когда его код уходит на сервер;
- когда он уходит во внешний AI;
- можно ли отключить AI;
- что будет, если AI ключ не настроен.

Нужен спокойный opt-in/notice, без пугающего текста.

#### 5. Довести "тренировку дня" до главного сценария

В интерфейсе уже есть много сильных частей: навыки, цели, daily, повторение, разборы. Теперь нужно собрать их в один главный путь:

1. Открыл приложение.
2. Видишь одну понятную кнопку: начать тренировку.
3. Получаешь 4-6 задач по слабым местам и повторению.
4. После ошибок видишь человеческое объяснение.
5. В конце получаешь короткий итог: что выросло, что повторить завтра.

Это важнее, чем добавлять еще 20 кнопок или новые режимы.

#### 6. Зафиксировать release checklist

Перед каждым релизом нужен короткий обязательный список:

- frontend contracts/smoke;
- backend build;
- backend audit;
- frontend audit;
- integration tests against staging;
- installer build;
- ручной запуск приложения;
- GitHub Release upload.

Сейчас проверки есть, но их надо собрать в один понятный ритуал.

### Что пока не нужно делать

#### Не добавлять новые языки

Сейчас уже есть JS, Python, Go, C, C++, C#, Java. Пользователю важнее качество обучения, объяснения и повторения, чем еще один язык.

#### Не строить большой marketplace задач

Это уводит продукт в сторону Codewars/LeetCode. Сила этого проекта в персональной тренировке, а не в огромной библиотеке чужих задач.

#### Не делать сложную социальную сеть

Друзья, группы, дуэли, команды и большой leaderboard можно отложить. Сначала одиночный учебный цикл должен быть сильным.

#### Не усиливать paywall раньше пользы

Если цель — продукт, который реально помогает, базовая учебная петля должна быть доступной. Pro можно держать для облака, AI, отчетов, синка, но нельзя ломать обучение.

#### Не делать AI автопилотом

AI должен объяснять, подсказывать и разбирать, но не превращать тренажер в "дай ответ". Главная ценность — человек учится думать.

#### Не начинать большой редизайн

Интерфейс уже движется в правильную сторону: меньше шума, больше фокуса. Лучше точечно полировать главный сценарий, чем снова перерисовывать все приложение.

#### Не пушить prod-интеграции без staging

Пока интеграционные тесты могут создавать пользователей и данные, их нельзя бездумно гонять по production как обычный smoke.

### Мой приоритетный порядок работ

1. Secure session storage в Electron.
2. Нормальный integration test runner со staging/local режимом.
3. Frontend audit risk по Monaco/DOMPurify.
4. Главный экран "тренировка дня" как основной путь.
5. Privacy/consent для AI.
6. Release checklist и автоматизация проверок.
7. Только потом — новые социальные/маркетинговые/Pro-фичи.

Этот документ не про монетизацию. Он про то, как сделать **JS Infinite Trainer** программой, которая реально помогает людям учиться, возвращаться к практике и получать максимум пользы от времени за кодом.

Главная мысль: продукт должен быть не "еще одним генератором задач", а личным тренером, который понимает слабые места пользователя, дозирует нагрузку, объясняет ошибки и превращает хаотичное обучение в понятный ежедневный маршрут.

## 1. Главная ценность продукта

Пользователь должен открывать программу и сразу понимать:

- что ему делать сегодня;
- почему именно эта задача полезна;
- какую слабость она закрывает;
- что улучшилось после решения;
- что повторить позже.

Идеальный опыт:

> "Я не выбираю, что учить. Я просто открываю тренажер, решаю 20-40 минут, и он сам ведет меня по моему уровню."

Это сильнее, чем просто "много задач". Людям нужны не задачи. Им нужен прогресс без хаоса.

## 2. Принципы, на которых надо строить продукт

### 2.1. Меньше выбора, больше направления

У новичков и джунов проблема не в нехватке материалов. Проблема в том, что материалов слишком много. Поэтому продукт должен выбирать за человека:

- следующую тему;
- сложность;
- формат тренировки;
- момент повторения;
- когда дать теорию, а когда код.

Настройки нужны, но основной путь должен быть "нажми Старт".

### 2.2. Ошибка — это главный учебный момент

Если человек ошибся, программа не должна просто писать "test failed". Нужно превращать ошибку в обучение:

- показать, какой тест упал;
- объяснить человеческим языком, что код сделал не так;
- подсветить тип ошибки: off-by-one, мутация данных, неверный тип, async timing, edge case;
- дать маленький намек, а не сразу решение;
- после исправления закрепить похожей задачей.

Это даст ощущение настоящего наставника.

### 2.3. Повторение важнее марафона

Продукт должен поощрять не "сидел 6 часов", а "возвращался регулярно и закрывал слабые места". Лучше 25 минут каждый день, чем редкие рывки.

В интерфейсе стоит показывать:

- "сегодня достаточно";
- "следующее повторение завтра";
- "ты закрыл слабую тему";
- "не дави, прогресс идет".

Это снижает выгорание.

### 2.4. Учить не только синтаксис, а мышление

По каждой задаче полезно фиксировать не только правильность, но и навык:

- декомпозиция;
- работа с краями;
- выбор структуры данных;
- чтение условия;
- асинхронное мышление;
- аккуратность с типами;
- тестирование гипотез.

Так продукт станет полезен не только новичкам, но и тем, кто уже пишет код.

## 3. Идеи фич, которые дадут максимум пользы

### 3.1. "Сегодняшняя тренировка"

Сделать главный режим, который сам собирает короткую сессию:

1. Разогрев: 1 легкая задача.
2. Основной блок: 2-4 задачи по слабым темам.
3. Повторение: 1 задача из старой темы.
4. Финал: короткий вывод, что улучшилось.

Важно: человек не должен думать, что выбрать. Он должен нажать одну кнопку.

Польза:

- меньше прокрастинации;
- понятный ритм;
- ощущение завершенной тренировки;
- проще возвращаться каждый день.

### 3.2. Диагностика уровня при первом запуске

Вместо пустого старта дать мини-диагностику на 10-15 минут:

- несколько задач по массивам, объектам, функциям, async, алгоритмам;
- оценить не только pass/fail, но и скорость, количество попыток, подсказки;
- после этого построить "карту навыков".

Результат:

- "Твой уровень: уверенный beginner / junior-ready / mid practice";
- "Сильные стороны";
- "Слабые темы";
- "План на 7 дней".

Это сразу дает пользователю ощущение персонализации.

### 3.3. Карта навыков

Сделать не просто XP, а карту:

- Массивы
- Объекты
- Функции
- Замыкания
- Async
- DOM
- Алгоритмы
- Чтение условий
- Edge cases
- Скорость решения

Каждый навык может иметь уровень:

- "не трогали";
- "начал";
- "закрепляется";
- "уверенно";
- "нужно повторить".

Польза:

- человек видит, что именно растет;
- легче понять, почему программа дает конкретную задачу;
- прогресс становится не абстрактным.

### 3.4. Умные подсказки по ступеням

Не давать ответ сразу. Сделать 4 уровня помощи:

1. Переформулировать задачу проще.
2. Подсказать идею.
3. Подсказать шаг алгоритма.
4. Показать почти-решение или разбор.

После использования подсказки XP можно снижать мягко, но не наказывать жестко. Подсказка — часть обучения.

### 3.5. Разбор после решения

После успешного решения показывать короткий блок:

- что ты применил;
- почему это решение работает;
- какие edge cases были важны;
- как можно решить иначе;
- что потренировать дальше.

Это важнее, чем просто зеленая галочка. Без разбора человек часто не понимает, что именно выучил.

### 3.6. Режим "объясни свое решение"

После задачи предложить пользователю в 1-2 предложениях объяснить подход. Можно не проверять строго, но сохранять.

Польза:

- человек учится формулировать мысли;
- это помогает собеседованиям;
- закрепляет материал лучше, чем пассивное чтение.

Позже сюда можно добавить AI-проверку объяснения.

### 3.7. Ошибки как отдельная история обучения

Сохранять типы ошибок:

- неверный return;
- мутировал входные данные;
- не обработал пустой массив;
- перепутал async;
- не учел отрицательные числа;
- неверно понял условие.

Потом программа может говорить:

> "Ты часто теряешь edge cases на пустых данных. Сегодня дадим 2 короткие задачи на это."

Это сделает тренажер намного умнее.

### 3.8. Повторение слабых тем

Уже есть SRS-идея, ее стоит сделать центральной:

- задача решена быстро и без ошибок — повторить позже;
- задача решена с трудом — повторить раньше;
- тема забыта — вернуть в тренировку;
- тема уверенная — не мучить человека лишним.

Польза: меньше ощущения "топчусь на месте", больше реального закрепления.

### 3.9. Персональные цели без давления

Сделать цели не только по streak:

- "15 минут в день";
- "3 задачи через день";
- "закрыть async за неделю";
- "подготовиться к junior interview";
- "вернуться после перерыва".

Разные люди учатся по-разному. Продукт должен уважать ритм человека.

### 3.10. Режим "вернулся после перерыва"

Если пользователь не заходил 7+ дней, не стыдить его. Показать:

- "Продолжим спокойно";
- легкий разогрев;
- восстановительную тренировку;
- повторение старых тем.

Это сильно повышает шанс, что человек не бросит продукт.

## 4. Идеи для продвижения через пользу, а не через рекламу

### 4.1. Публичные учебные челленджи

Не "купи Pro", а:

- "7 дней массивов";
- "14 дней JavaScript core";
- "30 дней до уверенного junior";
- "Async без боли";
- "Алгоритмы для тех, кто боится алгоритмов".

Каждый челлендж должен быть встроен в продукт и давать понятный результат.

### 4.2. Экспорт прогресса как учебный отчет

Сделать красивый отчет:

- сколько задач решено;
- какие темы улучшились;
- какие ошибки были частыми;
- что повторить дальше;
- какие задачи пользователь написал сам.

Такой отчет можно отправить ментору, другу или сохранить для себя.

Польза для продвижения: люди сами будут показывать прогресс.

### 4.3. "План на неделю" как шарящийся формат

После диагностики программа может генерировать:

> "Мой план на неделю: async, objects, edge cases. 25 минут в день."

Можно сделать экспорт в картинку или markdown. Это не агрессивная реклама, а полезный артефакт.

### 4.4. Библиотека разборов ошибок

Собрать в продукте и на GitHub/сайте базу:

- "Почему map возвращает undefined";
- "Как не забывать return";
- "Что такое edge case";
- "Почему async test падает";
- "Как читать условие задачи".

Это поможет пользователям даже без установки, а потом естественно приведет их к тренажеру.

### 4.5. Community без токсичности

Лидерборд может мотивировать, но может и давить. Нужен не только топ, а здоровая социальная механика:

- "люди твоего уровня";
- "сегодня тренировались 24 человека";
- "ты в группе тех, кто держит ритм";
- "поддержать друга";
- "совместная неделя без сравнения XP".

Цель — не сделать людей зависимыми от рейтинга, а помочь им не бросать.

### 4.6. Менторский режим

Для преподавателей, друзей и наставников:

- ученик экспортирует отчет;
- ментор видит слабые темы;
- ментор может назначить 5 задач;
- ментор пишет комментарий к прогрессу.

Это может стать одной из самых ценных функций, потому что многим нужен не курс, а сопровождение.

## 5. Что стоит улучшить в текущем продукте первым

### Приоритет 1: "Сегодняшняя тренировка"

Самый большой прирост пользы. Она превращает продукт из набора возможностей в понятный ежедневный ритуал.

Минимальная версия:

- кнопка "Начать тренировку";
- 5 задач;
- подбор по слабым темам;
- итоговый экран.

### Приоритет 2: Разбор ошибок

Сейчас проверка уже есть. Следующий шаг — объяснить человеку, почему он ошибся.

Минимальная версия:

- классифицировать ошибку по упавшему тесту;
- показать простое объяснение;
- предложить повторить похожий кейс.

### Приоритет 3: Карта навыков

XP мотивирует, но карта навыков объясняет прогресс.

Минимальная версия:

- 7 категорий;
- статус по каждой;
- "слабая тема дня".

### Приоритет 4: Custom tasks sync до идеального состояния

Если пользователь создает свои задачи, это его личная ценность. Их нельзя потерять.

Минимальная версия:

- синхронизировать создание;
- синхронизировать удаление;
- аккуратно мержить локальное и облачное;
- показывать статус синка.

### Приоритет 5: Нормальный onboarding

Первый запуск должен вести человека:

- выбрать цель;
- пройти мини-диагностику;
- получить первый план;
- решить первую задачу без перегруза интерфейсом.

## 6. Идеи, которые лучше не делать рано

### Не делать слишком рано: большой маркетплейс задач

Это уведет продукт в сторону Codewars. Главная сила сейчас — персональная генерация и адаптация.

### Не делать слишком рано: сложную социальную сеть

Друзья, дуэли и команды полезны, но только после того, как одиночный учебный цикл станет сильным.

### Не делать слишком рано: много Pro-ограничений

Если цель — польза, нельзя ломать обучение ради paywall. Все базовое обучение должно оставаться доступным.

### Не делать слишком рано: слишком много AI

AI полезен как наставник и разборщик ошибок, но не должен заменять самостоятельное мышление. Лучше давать подсказки ступенями.

---

## 7. Как понять, что продукт реально помогает

Нужны метрики пользы, а не только активности:

- сколько пользователей возвращаются на 2-й, 7-й, 30-й день;
- сколько задач решают без подсказок после повторения;
- уменьшается ли число одинаковых ошибок;
- растет ли скорость решения в той же теме;
- закрываются ли слабые категории;
- сколько людей экспортируют отчет или план;
- сколько людей создают свои задачи;
- сколько людей возвращаются после перерыва.

Главная метрика:

> Пользователь через месяц стал увереннее решать задачи и понимает, почему.

---

## 8. Идеальный путь пользователя

### День 1

1. Установил приложение.
2. Выбрал цель: "Хочу подтянуть JS".
3. Прошел диагностику.
4. Получил карту навыков.
5. Решил первую короткую тренировку.
6. Увидел понятный итог.

### День 2-7

1. Открывает "Сегодняшнюю тренировку".
2. Получает задачи по слабым темам.
3. Видит разбор ошибок.
4. Повторяет старые темы.
5. Получает план на завтра.

### Через месяц

1. Видит рост по карте навыков.
2. Понимает свои типичные ошибки.
3. Может экспортировать отчет.
4. Чувствует, что стал сильнее не по XP, а по реальному мышлению.

## 9. Тон продукта

Тренажер должен говорить как спокойный наставник:

- без токсичного "ты проиграл";
- без давления streak;
- без ощущения экзамена;
- с уважением к темпу человека;
- коротко, ясно, по делу;
- с поддержкой после ошибок.

Пример хорошего текста:

> "Тест упал на пустом массиве. Логика для обычных данных работает, осталось обработать крайний случай."

Пример плохого текста:

> "Wrong answer."

---

## 10. Самая сильная формула продукта

**Диагностика → персональная тренировка → разбор ошибки → повторение → карта роста.**

Если эта петля станет очень качественной, продукт будет нужен людям даже без агрессивного маркетинга. Он будет экономить им время, снижать хаос в обучении и давать ощущение настоящего прогресса.
